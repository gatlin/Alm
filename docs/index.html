Simple example: a little counter.

Alm's primary goal is to help you organize your applications. Apps do a lot of things: they handle browser events, manipulate the DOM, update state, and (try to) keep everything synchronized and consistent.

An App asks for the following right up front:

  • Your application state;
  • How to take an action and a state to update it;
  • How to render the view given a state;
  • Where in the document hierarchy to listen for events and render the view.

The alm.el function is how you can the parts of your view. It takes a tag name, an object of attributes, and an (optional) array of children, which can be either more alm.el calls, or plain strings for text nodes. 

                        
/* A simple counter */
const el = alm.el;
const app1 = new alm.App({
    eventRoot: 'app-1',
    domRoot: 'app-1',
    state: 0,
    update: (action, num) => num + (action ? 1 : -1),
    render: state =>
        el('div', { 'id':'app-1-main' }, [
            el('h3', { 'class':'app-1-header' }, [state.toString()]),
            el('span', {}, [
                el('button', { 'id':'up-btn' }, ['+']),
                el('button', { 'id':'down-btn' }, ['-'])
            ])
        ]),
    main: scope => {
        scope.events.click
            .filter(evt => evt.getId() === 'up-btn')
            .recv(evt => scope.actions.send(true));

        scope.events.click
            .filter(evt => evt.getId() === 'down-btn')
            .recv(evt => scope.actions.send(false));
    }
}).start();
